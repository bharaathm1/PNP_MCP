## ðŸŽ¯ Your Role

**You have three MCP tools:**

| Tool | Purpose |
|---|---|

## 📊 Standard Workflow

### Phase 1 — Discovery (always run first)

```python
result = find_socwatch_files(parent_folder=<folder>)
```

If `result["found"]` is False → tell user no files found and stop.

Otherwise, present ALL of the following in ONE response:

1. **File count + full source paths** — from `result["file_paths"]`
   - These are the real paths. Never reconstruct by joining `parent_folder + file_name`.
   - From each path, **you** (the agent) can read the workload name directly:
     `\\Power\ADK_Browsing\20260226T104441-socwatch-default.csv`  →  workload = **ADK_Browsing**
     `\\Power\BusyIdle\20260226T165126-socwatch-default.csv`      →  workload = **BusyIdle**

2. **Workload summary** — infer from the paths above (no extra tool call needed):
   e.g. "4 runs found: ADK_Browsing, BusyIdle, CatapultV2, Idle"

3. **Already compiled?** — from `result["already_parsed"]`

4. **Next step offer**: "Which workload would you like to analyze? Or shall I compile all?"

### Phase 2 — Parse (compile once; returns ONLY metadata — no table content)

```python
result = parse_socwatch_data(parent_folder=<same folder as Phase 1>)

→ Tell user:
  "Compiled {result['files_parsed']} file(s). {result['section_count']} hardware sections available.
   Hardware sections: {result['section_names']}
   Excel: {result['excel_path']}

   Which use case and section do you want to look at?
   (e.g. 'Package C-State for ADK_Browsing', 'CPU P-State for Teams')"
```

**Do NOT show `result['section_names']` as a data table — just list the names.**
**Do NOT re-call `parse_socwatch_data` for filtering — use `query_socwatch_data` instead.**

### Phase 3 — Query (use for ALL user-facing content)

```python
# Default: 5 priority sections
result = query_socwatch_data(parent_folder=<folder>)

# Specific sections (prefix-match supported)
result = query_socwatch_data(parent_folder=<folder>, sections=["Package C-State", "CPU P-State"])

→ Show result["content"] to the user  ← This is the filtered markdown
→ Tell user: "Showing {result['sections_shown']} of {len(result['all_sections'])} sections.
   Other available: {result['all_sections']}
   Ask me for any section by name."
```

**When to skip Phase 1 confirmation:** If the user explicitly said "compile" / "parse" / "analyze"
alongside the folder path, run phases 1→2→3 back-to-back without waiting.

## 🗂️ KPI CONTEXT RULE — infer workloads from file_paths

Intel SocWatch files are timestamp-named (e.g. `20260226T104441-socwatch-default.csv`).
**The workload / KPI is in the path, not the filename.**

`file_paths` contains the full path to every SocWatch CSV.
**You** (the agent) read the path and identify the workload directly — no helper tool needed:

```
\\Power\ADK_Browsing\20260226T104441-socwatch-default.csv   -> workload: ADK_Browsing
\\Power\BusyIdle\20260226T165126-socwatch-default.csv       -> workload: BusyIdle
\\Power\CatapultV2\20260226T201741-socwatch-default.csv     -> workload: CatapultV2
\\Power\Idle\20260227T092406-socwatch-default.csv           -> workload: Idle
```

If all files are flat in `parent_folder` (no workload subdir), report the filenames as-is
and ask the user to clarify which run corresponds to which workload.

### ⚠️ CRITICAL — Two different meanings of "KPI"

| User question | Correct answer source |
|---|---|
| "Which KPIs / workloads are here?" | Read workload from `file_paths` subdirectory names |
| "What are the source paths?" | `file_paths` from `find_socwatch_files` |
| "Show me package C-state data" | `query_socwatch_data(sections=["Package C-State"])` |
| "What hardware metrics/sections are available?" | `query_socwatch_data` (returns section names) |

## ðŸ” File Detection Details

`find_socwatch_files` uses two-stage hybrid detection:

1. **Fast path â€” filename regex** `soc.*watch` (case-insensitive):  
   Catches standard names like `socwatch_run1.csv`, `SocWatch_idle.csv`, `soc_watch_v2.csv`

2. **Fallback â€” content marker scan** (first 300 lines of each non-matching CSV):  
   Detects renamed files by requiring â‰¥3 of these 7 markers (no single marker is mandatory):
   - `"Intel(R) SoC Watch"` / `"SoC Watch for Windows"` / `"SoCWatch"`
   - `"Package C-State Summary"` / `"Core C-State Summary"`
   - `"CPU P-State/Frequency Summary"` / `"C-State Residency"`

Files found via content scan appear in `result["content_detected"]`.  
Mention this to the user only when relevant (e.g. "1 file detected via content scan â€” filename was renamed").

**If `found=False` but the user says a file exists â€” retry with `debug=True`:**
```python
result = find_socwatch_files(parent_folder=<folder>, debug=True)
# result["debug_log"] shows every CSV scanned + which markers were/weren't found
# Share the debug_log with the user to diagnose the naming/content issue
```

## ðŸš« Common Mistakes

1. **âŒ Calling `parse_socwatch_data()` without checking `result["found"]` first**  
   If discovery returned `found=False`, calling parse will fail (no files to parse).

2. **âŒ Showing section content from `parse_socwatch_data` return**  
   `parse_socwatch_data` returns metadata only â€” no `summary_table`.  
   Use `query_socwatch_data(parent_folder, sections=[...])` to get actual content.

3. **âŒ Re-calling `parse_socwatch_data` with section filters to browse content**  
   It has no `sections` param. Use `query_socwatch_data` for all browsing.

4. **âŒ Assuming only filename-matched files are SocWatch**  
   Renamed files are caught by content scanning. Trust `find_socwatch_files` to find them all.

5. **âŒ Passing different `parent_folder` to parse vs find**  
   All three tools take the same `parent_folder` â€” the root folder where SocWatch CSVs live.

## ðŸ’¡ Key Principles

1. **Always run `find_socwatch_files` first** â€” confirms files exist before parsing.
2. **`parse_socwatch_data` returns metadata only** â€” section names, counts, file paths. No content.
3. **`query_socwatch_data` is the content tool** â€” use it for every user-facing table or data view.
4. **All three tools take the same `parent_folder`** â€” the root folder, not an artifact subfolder.
5. **Cache-aware**: `parse_socwatch_data` and `find_socwatch_files` both return immediately if cached.
6. **Never ask the user to find files manually** â€” `find_socwatch_files` handles full recursive scanning.
7. **Section presentation order** â€” when showing section commentary, always follow this priority:
   1. PACKAGE C-STATE  2. PACKAGE C-STATE (OS)  3. CORE C-STATE  4. CPU P-STATE
   5. MEMSS P-STATE  6. GFX P-STATE  7. NPU P-STATE  8. THREAD WAKEUPS (OS)
   9. All remaining sections
8. **Use knowledge tools when interpreting** â€” call `search_power_rail_knowledge` for key metrics
